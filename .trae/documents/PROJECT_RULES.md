# 墨灵项目协作规则

## 📋 核心工作流规则

### 1. 代码推送规则
**重要：所有代码变更必须经过以下流程才能推送到 GitHub main 分支！**

```
开发阶段 → 本地测试 → 云端验证（GitHub Actions）→ 用户确认 → 推送代码
```

#### 详细步骤
1. **开发阶段** - 在本地进行开发和修改
2. **本地测试** - 确保：
   - `pnpm run build` 构建成功
   - 运行 `pnpm run dev` 并验证功能正常
3. **云端验证** - （已配置 GitHub Actions 自动触发）
4. **用户确认** - 必须经过用户同意后才能推送
5. **最后推送** - 用户确认无误后再执行 `git push`

### 2. UI/UX 改进规则
- 每次视觉样式变更后，必须：
  - 在本地开发模式下预览效果
  - 验证颜色、间距、动画等视觉细节
  - 确认交互流畅无卡顿

### 3. 构建与测试
- 任何代码变更前，确保有完整的 Git 提交历史
- 保持 main 分支始终处于可构建状态
- 构建失败时，先修复问题再继续

## 🎨 视觉风格指南

### 水墨东方美学系统
- **墨黑层次**：ink-900 > ink-800 > ink-700 > ink-600 > ink-500
- **强调色**：
  - 朱砂红（cinnabar）- 主要强调、按钮、激活状态
  - 竹青（bamboo）- 次要强调
  - 藤黄（rattan）- 辅助提示
- **主文字**：paper（宣纸色）
- **次要文字**：paper-dim
- **字体**：
  - 标题：Noto Serif SC（衬线）
  - 正文：Noto Sans SC（无衬线）

## 📦 技术约束

- 使用 pnpm 作为包管理器
- 严格遵守 TypeScript 类型约束
- TailwindCSS 优先，内联样式为辅
- 保持现有功能完整性，除非有明确改进理由

## 🔄 回滚策略
- 任何变更都可以通过 Git 回滚
- 遇到破坏性问题时优先考虑 `git revert` 而非 `git reset`
